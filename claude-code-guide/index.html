2024년 초의 이미지. 정의는 "말 잘하는 챗봇". 지난 2년 실체는 두 번 바뀌고 속도는 한 번 더 가팔라짐. 이 장은 그 2년, 그리고 오늘부터의 지표.

Introduction

바이브 코딩이란?

코드를 직접 쓰지 않고, 자연어 대화로 소프트웨어를 만드는 방식.

바이브 코딩의 스펙트럼

넓게 보면 세 단계. 왼쪽일수록 사람이 키보드, 오른쪽일수록 AI가 실행. 이 책에서 말하는 바이브 코딩은 오른쪽 끝 — 에이전트 루프.

Level 1

자동완성

Copilot · Tabnine · Cursor Tab

Human 85%AI 15%

다음 몇 줄 제안 → 사람이 수락·거부 반복. 파일 단위 설계는 전적으로 사람.

Level 2

페어 프로그래밍

Cursor Composer · ChatGPT · Windsurf

Human 50%AI 50%

대화로 블록 생성 → 사람이 검토·붙여넣기. 파일 몇 개 편집은 AI, 실행·검증은 사람.

Level 3

이 책

에이전트 루프

Claude Code · Cursor Agent · Aider

Human 20%AI 80%

목표·제약만 주면 스스로 파일 읽고·수정·실행·테스트. 사람은 방향 교정과 결과 검증.

비교: 기존 방식 vs 바이브 코딩

기존 방식
1. 문법 검색...
2. 코드 작성...
3. 에러 디버깅...
4. Stack Overflow...
⏱ 수 시간
바이브 코딩
❯ CSV 파일 읽어서 bar chart로 시각화해줘
↳ Read data.csv
↳ Write plot.py
↳ Bash: python plot.py
✓ chart.png 생성 완료 (30초)

운전석이 바뀝니다

코드 쓰는 사람 → 의도를 전달하고 결과를 검증·통합하는 사람. 속도가 오른 게 아니라, 작업의 성격이 달라짐.

CLAUDE.md

매 세션마다 자동 로드되는 프로젝트 설명서. 프로젝트가 Global을 override 합니다.

Global · managed settings
User · ~/.claude/
Project
./CLAUDE.md

안쪽이 바깥을 override — Project가 가장 우선.

왜 써야 하는가?

매 세션 Claude는 아무것도 모르고 시작합니다. 지난 대화의 규칙, 프로젝트의 관습, 건드리면 안 되는 영역 — 전부 재설명해야 하죠. CLAUDE.md는 그 재설명을 파일로 못 박아 두는 장치입니다.

01 · Context

매번 설명하지 않아도 됨

빌드 명령·스택·폴더 구조·컨벤션을 세션마다 다시 가르칠 필요가 없습니다. 파일을 열자마자 Claude가 "이 프로젝트는 uv + Hydra"를 알고 시작.

02 · Guardrail

실수 방지 레일

pip 대신 uv, main에 직접 push 금지, raw 데이터 read-only — 자율 모드에서도 지켜야 할 선을 코드로 못 박습니다.

03 · Reproducibility

팀·미래의 나와 공유

git에 커밋되므로 동료의 Claude도 같은 규칙을 따릅니다. 3개월 뒤 돌아와도 "왜 이렇게 짰지?"가 파일로 남아 있음.

잘 쓴 CLAUDE.md의 원칙

✓ 구체적 명령·경로·금지사항

"make train CFG=..." 같은 실행 가능한 문장. 추상적 선언 대신 Claude가 그대로 복붙할 수 있는 형태로.

✗ 장황한 아키텍처 설명

코드·README에서 얻을 수 있는 정보는 중복. CLAUDE.md는 Claude의 행동을 바꾸는 규칙만.

실전 예시 — DL 연구 프로젝트

p-tau217 혈장 바이오마커로 뇌 tau PET SUVR을 예측하는 멀티모달 딥러닝 파이프라인. 데이터 거버넌스·실험 재현성·팀 협업 규칙을 한 파일로.

md ptau217-tau-pred/CLAUDE.md
# ptau217-tau-pred
 
혈장 p-tau217로 뇌 tau PET SUVR을 예측하는 멀티모달 DL 파이프라인.
데이터: ADNI3 · A4 · 자체 코호트 (n=1,842, APOE ε4 층화 분할).
 
## Stack
- Python 3.11 · PyTorch 2.3 · MONAI 1.4 · Hydra · Weights & Biases
- 패키지 관리는 uv만 사용 — pip · conda 직접 호출 금지
- 실험 추적: wandb (entity=neurolab, project=ptau217)
 
## Commands
- make setup: uv sync + pre-commit install
- make train CFG=configs/baseline.yaml: 단일 GPU 학습
- make sweep: wandb sweep (hydra multirun)
- pytest -q: 단위 테스트 (모델·데이터로더)
- ruff check . && mypy src: 린트 + 타입 체크
 
## Code conventions
- 모듈 진입점은 src/ptau217/ — 루트에 새 .py 추가 금지
- 설정은 Hydra yaml로만 — argparse · 하드코딩 하이퍼파라미터 금지
- 난수 시드 42 고정 · torch.use_deterministic_algorithms(True)
- tensor shape는 einops 주석 또는 assert로 명시
- 타입 힌트 필수 (mypy strict 통과)
 
## Data governance
- 원본은 /data/raw/ — read-only mount, 절대 수정 금지
- 파생물은 /data/processed/{dataset}/{version}/ 에만 저장
- PHI 필드(MRN · DOB · 이름)는 로그 · wandb · 커밋에 절대 남기지 말 것
- 환자 분할은 split_by_subject()만 사용 (site/visit 기반 분할은 leakage)
 
## Experiment rules
- 체크포인트는 artifacts/{run_id}/ — wandb artifact로만 공유
- 모든 run은 commit hash + config hash를 wandb config에 기록
- 사전학습 가중치 교체·데이터 버전 변경은 CHANGELOG.md에 기록
 
## Git workflow
- main 직접 push 금지 — feature 브랜치 → PR
- 커밋 메시지: type(scope): 한글 요약 (예: feat(model): cross-attn fusion 추가)
- notebooks/ 커밋 전 nbstripout 필수 (출력 포함 시 PHI 누출 위험)
 
## Don't
- raw DICOM/NIfTI에 덮어쓰기 금지
- CUDA OOM 시 코드에서 batch_size 변경 금지 — config 또는 grad_accum로만
- 실패한 실험 삭제 금지 — wandb에 tags=[failed]로 표시
- 테스트 셋 성능은 최종 제출 전까지 한 번만 평가

이 정도가 되면 Claude는 pip install을 제안하지 않고, main에 직접 커밋하지 않고, wandb 로그에 환자 ID를 남기지 않습니다. 규칙을 만들어둔 만큼 자율 모드가 안전해집니다.

서브에이전트가 참조하는 전용 지침 파일.

메인 CLAUDE.md와 별도로, 에이전트별 행동 규칙 설정

Skill의 핵심 축은 "간단 / 고급"이 아니라 "누가 호출하는가". description이 맞으면 Claude가 알아서 꺼내 쓰고, 프론트매터 한 줄로 수동 전용 제한도 가능.

Default · Claude + User

자동 호출 허용

description 매칭으로 Claude가 필요할 때 스스로 꺼내 씀. 사용자도 /name으로 직접 호출 가능.

용도: 코딩 컨벤션 · 리뷰 체크리스트 · 도메인 레퍼런스

User Only

수동 호출 전용

disable-model-invocation: true — side effect 있거나 타이밍 제어가 필요한 작업. Claude가 멋대로 돌리면 안 되는 것.

용도: /commit · /deploy · /send-slack · /release

자동 호출 — 어떻게 고르나

사용자 발화가 들어오면 Claude가 어떤 skill을 쓸지 결정하는 순서. description이 매칭 키 — 잘 쓰면 놓치지 않고, 막연히 쓰면 안 불린다.

Step 01 · 세션 시작

등록

모든 SKILL.md의 name + description이 Claude 컨텍스트에 올라감. 본문은 아직 로드 X — token 절약.

Step 02 · 발화 입력

의미 매칭

사용자 프롬프트 ↔ 각 skill의 description 의미 매칭. 적합하면 후보 선정, 여러 개면 가장 적합한 것.

Step 03 · 호출

전체 로드 · 실행

선정된 skill의 SKILL.md 전체가 컨텍스트에 들어오고 그 지시대로 실행. 세션 동안 유지.

auto-invoke · 예시 발화
❯ 새 실험 스크립트 하나 만들어줘. CIFAR-10에 resnet18 돌리는 거
↳ [matched] /experiment-conventions
↳ description: "PyTorch 실험 스크립트 규칙. 학습 코드 작성 시 사용 ..."
→ seed 고정 · W&B 로깅 · checkpoint 2개 보관 규칙 적용해서 작성

SKILL.md 예시 2종

모두 ~/.claude/skills/<이름>/SKILL.md에 저장하면 /<이름>으로 호출됨. frontmatter가 invocation·도구·서브에이전트·인수를 제어.

01 · 참조 · 자동 호출

experiment-conventions

PyTorch 실험 컨벤션. 학습 스크립트 작성 중 Claude가 알아서 참조.

md skills/experiment-conventions/SKILL.md
---
name: experiment-conventions
description: PyTorch 실험 스크립트 규칙 — 학습 코드 작성·수정·리뷰 시 사용. seed · W&B · checkpoint · reproducibility 표준.
---
 
실험 스크립트 작성 시:
- Seed 고정: seed_everything(cfg.seed)
  (torch · numpy · random 모두)
- W&B 로깅: wandb.init(project, config=cfg)
- Checkpoint: epoch별 + best metric 2개 보관
- Config: Hydra (conf/*.yaml)
- Device: torch.device(cfg.device), autodetect 금지

02 · 태스크 · 수동 전용

submit-job

SLURM 잡 제출. Compute 비용 발생 — Claude 자동 invoke 금지.

md skills/submit-job/SKILL.md
---
name: submit-job
description: SLURM 잡 제출 — compute 비용 발생, 반드시 사람 확인
disable-model-invocation: true
argument-hint: [config-path]
allowed-tools: Bash(sbatch *) Bash(squeue *) Read
---
 
SLURM 잡 제출 절차:
1. $ARGUMENTS config 읽어서 요약
  (model · dataset · lr · bs · gpus)
2. 예상 runtime · GPU 시간 리포트
3. 사용자 확인 받은 뒤 sbatch 실행
4. squeue로 job ID 반환

실행 예시 · /submit-job 수동 호출

~/experiments
❯ /submit-job conf/resnet18_cifar10.yaml
↳ Read: conf/resnet18_cifar10.yaml
Config 요약:
model=resnet18 · dataset=CIFAR-10 · lr=1e-3 · bs=128 · gpus=2
예상 runtime: ~45분 · 2× A100 (GPU시간 1.5h)
제출할까요? (y/n)
❯ y
↳ Bash: sbatch scripts/launch.slurm conf/resnet18_cifar10.yaml
✓ Submitted batch job 831427

하네스 엔지니어링
자동화, 도구, 안전장치

프롬프트 엔지니어링은 "이번 대화에서 어떻게 지시할까"에 집중합니다. 하네스 엔지니어링은 한 단계 더 나아가, 매번 좋은 프롬프트를 치지 않아도 AI가 알아서 잘 동작하도록 환경 자체를 설계합니다.

좋은 하네스의 3원칙

1

반복 제거

매번 같은 지시를 반복하지 않도록
CLAUDE.md와 Skills로 고정

2

안전장치

위험한 명령 차단, 시크릿 유출 방지
Hooks로 자동화된 가드레일

3

팀 공유

프로젝트 .claude/ 폴더를 git에 커밋
팀 전체가 같은 하네스 사용

실제 하네스 구성 예시

tree project/.claude/
# 프로젝트 루트의 .claude/ 폴더
├── CLAUDE.md # 프로젝트 규칙, 빌드 명령
├── settings.json # hooks, MCP 서버 설정
├── commands/ # 커스텀 슬래시 커맨드
│ ├── commit.md
│ └── review.md
├── skills/ # 고급 스킬
│ └── deploy/
│ └── SKILL.md
└── agents/ # 커스텀 에이전트
├── code-reviewer.md
└── debugger.md

하네스가 동작하는 모습

~/my-research-project
$ claude
Reading CLAUDE.md... ← 프로젝트 규칙 로드
Loading 3 MCP servers... ← 외부 도구 연결
Loading 2 custom agents... ← 에이전트 준비
SessionStart hooks executed... ← 자동화 실행
❯ 실험 데이터 분석하고 결과 시각화해줘
↳ MCP: PostgreSQL 쿼리로 데이터 조회
↳ Agent: code-reviewer가 분석 코드 검증
↳ Write: plot.py 생성
↳ Bash: python plot.py
↳ Hook: PostToolUse → 코드 포맷팅 자동 실행
✓ results/ 폴더에 분석 결과 + 시각화 저장 완료
❯ /commit
↳ Skill: 변경사항 분석 → 커밋 메시지 생성
↳ Hook: PreToolUse → 시크릿 스캔 통과 ✓
✓ feat: 실험 데이터 분석 및 시각화 추가

핵심 인사이트

하네스 엔지니어링의 목표는 "claude 하나만 치면 모든 것이 준비되는 환경"을 만드는 것입니다.

한 번 잘 설계해두면, AI는 매번 프로젝트 규칙을 알고, 안전장치 안에서 작동하며, 필요한 도구에 접근하고, 반복 작업을 자동으로 처리합니다.
